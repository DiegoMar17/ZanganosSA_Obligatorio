## 1. Cards de KPIs de empleados — bug visual

**Síntoma reportado:** las cards que muestran cuántos empleados hay registrados se veían como "una cadena de strings plana".

**Causa raíz:** `Views/Equipo/Index.cshtml` usaba las clases `kpi-row`, `kpi-card`, `kpi-label`, `kpi-val`, que nunca se definieron en `site.css` — no existían en ningún lado del proyecto.

**Fix:** se reemplazaron por el sistema de cards ya existente y usado en el resto de la app (`g3` + `stat-card` + `stat-card-accent` + `stat-lbl` + `stat-val` + `stat-meta`, el mismo patrón del Dashboard y Finanzas), con ícono, color y grilla de 3 columnas.

## 2. Perfil de empleado para admin

**Pedido:** el admin debe poder entrar directamente al perfil de un usuario, ver a qué equipo/sector pertenece, y ver sus tareas asignadas, en proceso o completadas.

**Decisión de modelo de datos:** `Tarea` solo tiene el booleano `Completada` (no existe un estado real de "en proceso" en ningún lado del código). Se le preguntó al usuario si quería agregar un estado real (`pendiente/en_proceso/completada`, con migración) o mantenerlo simple. El usuario eligió **mantener el modelo de 2 estados existente**, mostrando "Asignadas" (total), "En proceso" (pendientes) y "Completadas" derivadas del booleano actual.

**Implementación:**
- `EquipoController.cs`: nueva acción `Detalle(string id)` — carga el empleado, su apiario asignado, sus tareas, registros de auditoría de la semana y último acceso exitoso. Nuevo `EmpleadoDetalleViewModel` con `TareasAsignadas`, `TareasPendientes`, `TareasCompletadas`.
- `Views/Equipo/Detalle.cshtml` (nuevo): card de perfil, card de actividad (sector con link a `Apiarios/Detalle`, registros de la semana, último acceso), 3 `stat-card` de tareas, y listado de tareas reutilizando los estilos `task-row`/`t-info`/`chip` de Planificación.
- `Views/Equipo/Index.cshtml`: cada card de empleado ahora tiene un link "👤 Ver perfil →" hacia `Equipo/Detalle`.

**Recomendaciones dadas para "mejoras de empleado"** (solo sugeridas, no implementadas — el usuario pidió recomendación, no código): auto-servicio de su propio resumen de tareas en Perfil, historial de accesos propio, integrar notificación de tarea nueva al panel de notificaciones, avatar con iniciales en vez de emoji fijo, filtro/paginado en "Tareas y visitas" cuando la lista crece.

## 3. Admin puede tachar tareas de sus empleados

**Pedido:** "como administrador, quiero poder tachar las tareas de mis empleados [...] soy el que tiene el control total de la aplicación web".

Nota: el admin ya podía hacerlo desde `Planificación` (esa vista siempre mostró todas las tareas al admin, con checkbox). Lo que faltaba era la misma capacidad desde el perfil individual del empleado (`Equipo/Detalle`), donde el checklist era de solo lectura.

**Fix:**
- `EquipoController.cs`: nueva acción `CompletarTarea(int id, string userId)` — togglea `Completada`, registra en `Auditoria` vía `AuditoriaService` (mismo patrón que `PlanificacionController`), y redirige de vuelta a `Detalle` del mismo empleado.
- `Views/Equipo/Detalle.cshtml`: el checkbox de cada tarea pasó de un `<div>` estático a un `<form>` que postea a `CompletarTarea`.

## 4. Bug de UX: la barra lateral vuelve arriba al navegar

**Pedido:** al seleccionar cualquier función de la barra lateral, no debe "subir de vuelta" — si "Ver perfil" (al final del menú) estaba visible, debe seguir visible tras navegar a otra sección, sin tener que desplazarse de nuevo.

**Causa:** es una app MVC con recarga completa de página en cada navegación; el `<nav class="sb-nav">` (con `overflow-y:auto`) vuelve a `scrollTop = 0` en cada request.

**Fix** (`wwwroot/js/site.js`): se guarda el `scrollTop` de `.sb-nav` en `sessionStorage` en cada evento `scroll`, y se restaura al cargar la página.

---

## 5. Bug de UX: label "Tareas y visitas" se rompe

**Pedido:** "el nombre de la función 'Tareas y visitas' está pasándose mal, trata de cambiarlo/repararlo".

**Causa raíz:** el texto de cada `.sb-item` era un nodo de texto suelto dentro de un contenedor flex sin `white-space`/`overflow` definidos. Con el label largo ("Tareas y visitas") más el badge numérico de tareas pendientes, el conjunto no entraba en los 236px de ancho de la barra y el texto se partía en dos líneas, rompiendo el ítem — solo se manifestaba cuando el empleado tenía tareas pendientes (con badge visible).

**Fix:**
- `wwwroot/css/site.css`: nueva clase `.sb-label { flex:1 1 auto; min-width:0; overflow:hidden; text-overflow:ellipsis; white-space:nowrap }`, y `flex-shrink:0` agregado a `.sb-badge`.
- `Views/Shared/_Layout.cshtml`: se envolvió el texto de **todos** los ítems del sidebar (no solo el afectado) en `<span class="sb-label">`, para prevenir que el mismo bug aparezca en otro ítem si en el futuro se le agrega un badge.

## 6. Falsa alarma de "errores de compilación"

El usuario reportó errores de compilación y pidió revisar el proyecto y eliminar código innecesario. Al investigar:

- `dotnet build` fallaba con `MSB3027`/`MSB3021` — no eran errores de C#/Razor, sino que el archivo `ColmenaEmpresa.exe` estaba bloqueado por una instancia de la app corriendo en background (`PID 12104`).
- Se confirmó con el usuario antes de actuar; se cerró el proceso (`taskkill /F`) y se volvió a compilar: **0 errores, 0 advertencias**.

**Revisión de código muerto:** se usó un subagente Explore para auditar Controllers/Models/Views/Services/Data buscando `using` sin usar, métodos sin llamar, código comentado, prints de debug y propiedades de modelo sin referencia. No se encontró código muerto significativo — el proyecto está limpio.

**Hallazgo real de espacio, no resuelto por pedido del usuario:** 785 archivos de `bin/`, `obj/`, `publish/` y `colmena.db` quedaron commiteados en git antes de que existiera el `.gitignore` que ya los excluye (~132MB extra en el repo, 37MB en `.git`), más un archivo suelto `!DOCTYPE htmlAPiculTorp.txt` (122KB, mockup HTML viejo sin referencias) en la raíz del proyecto. El usuario decidió explícitamente **no tocar el repo/git** por ahora — queda documentado como pendiente.
